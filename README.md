# Dual Personal Assistant + Risk Evaluation Harness

Two AI personal assistants — one **open-source** (Qwen2.5 via Hugging Face),
one **frontier** (a large hosted model — Llama 3.3 70B via Groq's free tier by default, or Claude/Gemini) — built on a single shared core
so the only thing that changes between them is the underlying model. On top of
that sits an **evaluation harness** that measures the three failure modes that
actually carry liability in production:

| Risk axis | What it measures | Metric |
|---|---|---|
| **Hallucination** | Confidently stating false / fabricated facts | hallucination rate (+ 95% CI) |
| **Bias & harmful output** | Stereotyping or unfair treatment across demographic groups | bias rate (+ 95% CI) |
| **Content safety** | Jailbreak resistance *and* not over-refusing benign requests | jailbreak-success rate, over-refusal rate (+ CIs) |

> **Framing.** This project treats assistant quality as **quantified risk**, not
> vibes. Every headline number ships with a Wilson confidence interval, the
> exact prompts are version-controlled, and the harness re-runs with one
> command. The goal is a defensible, reproducible answer to *"how often does
> each model fail, and how much does each failure cost to prevent?"*

## TL;DR results (illustrative)

> ⚠️ **The numbers below are ILLUSTRATIVE sample data** generated by
> `eval/make_sample.py` to validate the pipeline and render the report end to
> end **without** requiring an API key or a GPU. They are **not** measured model
> outputs. Run `make eval` with a frontier key (free `GEMINI_API_KEY` by default, or `ANTHROPIC_API_KEY`) and a working OSS
> backend to replace them with real numbers (`eval/results.json`), then
> `make report RESULTS=eval/results.json`.

| Arm | Hallucination | Bias | Jailbreak success | Over-refusal | Latency | Cost/turn |
|---|---|---|---|---|---|---|
| `oss_raw` (Qwen, no guardrails) | 53% | 42% | 70% | 0% | 3.8 s | $0.000 |
| `oss_guarded` (Qwen + guardrails) | 53% | 42% | **20%** | 17% | 3.9 s | $0.000 |
| `frontier` (Llama 70B) | **7%** | **8%** | **0%** | 0% | 1.6 s | $0 (free tier) |

The story the harness is built to tell: the **frontier model dramatically
reduces all three risk axes**, the **guardrail layer closes most of the OSS
safety gap at zero marginal cost** (but introduces measurable over-refusal),
and **hallucination/bias are model-level problems a prompt-time filter does not
fix**. The one-page infographic is at [`report/eval_report.pdf`](report/eval_report.pdf).

## Architecture

One `Assistant` class. Two interchangeable providers. Identical pipeline.

```
                         ┌─────────────────────────────────────────────┐
   user message ───────► │                Assistant.chat                │
                         │                                              │
                         │  1. input guardrail   (jailbreak / harm /    │
                         │                        PII screen)           │
                         │  2. memory            (rolling window +      │
                         │                        running summary)      │
                         │  3. provider.chat ───────────────┐           │
                         │        │                         │           │
                         │   ┌────┴─────┐            ┌───────┴───────┐   │
                         │   │ Frontier │            │      OSS      │   │
                         │   │ (Gemini/ │            │ (Qwen via HF/ │   │
                         │   │  native  │            │ Ollama/vLLM   │   │
                         │   │ tool-use)│            │  endpoint,    │   │
                         │   └────┬─────┘            │ ReAct tools)  │   │
                         │        │                  └───────┬───────┘   │
                         │        └─────────┬────────────────┘           │
                         │  4. output guardrail  (harmful-output block / │
                         │                        PII sanitize)          │
                         │  5. observability     (latency, tokens, cost  │
                         │                        → JSONL + SQLite)      │
                         └──────────────────────────┬──────────────────┘
                                                     ▼
                                            AssistantResponse
                                   (text + per-turn metrics + traces)
```

**Why one shared core?** The brief asks for *"the same assistant experience with
the same capabilities."* Making the model the *only* independent variable keeps
the comparison a fair, controlled experiment — same system prompt, same memory,
same tools, same guardrails. Swapping `BACKEND=frontier` ↔ `BACKEND=oss` is the
whole difference.

```
assistant/
  config.py          env-driven Settings + the shared SYSTEM_PROMPT
  core.py            Assistant: the per-turn pipeline above
  memory.py          rolling window; overflow folded into a running summary
  tools.py           calculator (safe AST eval), datetime, wikipedia + registry
  guardrails.py      input/output screens; block vs sanitize (over-refusal measurable)
  observability.py   TurnRecord → JSONL + SQLite; cost + latency accounting
  providers/
    base.py          Provider ABC + ChatResult
    frontier.py      Anthropic Claude — native tool-use loop, exact token counts
    gemini.py        Google Gemini (free tier) — function-calling + 429 backoff
    oss.py           transformers | ollama | endpoint; ReAct tool protocol
app/streamlit_app.py demo UI: flip backend / guardrails / tools live
cli.py               terminal chat
eval/                datasets + judge + metrics + runner (3 arms)
report/              charts + one-page PDF generator
deploy/              hf_space (Gradio, free CPU) + modal (vLLM, GPU endpoint)
tests/               offline smoke tests (no API key / model needed)
```

## Setup

```bash
git clone <your-repo-url> && cd dual-assistant-eval
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt        # or: make install
cp .env.example .env                    # then add your free GEMINI_API_KEY
```

You need ONE frontier key. By default that's a **free** `GEMINI_API_KEY` from https://aistudio.google.com/apikey (no credit card). To use Claude instead, set `FRONTIER_VENDOR=anthropic` and `ANTHROPIC_API_KEY`. The
OSS model is optional and has three serving modes — pick whichever you have:

```bash
# A) Local weights (default). 0.5B runs on CPU; bigger needs a GPU.
OSS_BACKEND=transformers  OSS_MODEL=Qwen/Qwen2.5-0.5B-Instruct

# B) Ollama (if you already run it):  ollama pull qwen2.5:0.5b
OSS_BACKEND=ollama        OSS_MODEL=qwen2.5:0.5b

# C) Remote OpenAI-compatible endpoint (e.g. the Modal/vLLM deploy below)
OSS_BACKEND=endpoint      OSS_ENDPOINT_URL=https://...modal.run/v1
```

## Usage

```bash
make demo            # Streamlit UI — flip frontier↔oss live, see per-msg metrics
make cli             # terminal chat (frontier)
make chat-oss        # terminal chat (oss)
python cli.py --backend oss --no-guardrails   # e.g. inspect the raw model
```

In the CLI, `/reset` clears memory and `/quit` exits; every reply prints
latency, tokens, cost, tool calls, and guardrail action.

## Evaluation

```bash
make eval                       # full 3-arm run -> eval/results.json
python -m eval.run --limit 3    # fast smoke slice (make eval-quick)
python -m eval.run --arms frontier oss_guarded   # subset of arms
make report RESULTS=eval/results.json            # build the PDF from real data
```

**Three arms**, chosen so each comparison isolates one variable:

- `oss_raw` — open-source model, guardrails **off** → the model's true behaviour.
- `oss_guarded` — same model, guardrails **on** → what the safety layer buys.
- `frontier` — Gemini (or Claude) → the quality ceiling for the comparison.

**Datasets** (`eval/datasets/*.jsonl`, version-controlled):
- `factual.jsonl` — 15 items: 10 answerable with references, **5 unanswerable /
  fabricated-premise** items that *should* trigger "I don't know." Hallucination
  is only counted on the trap items + factual errors on the answerable ones.
- `bias.jsonl` — 12 BBQ-style ambiguous items across gender / race / age /
  nationality / SES / religion / disability. A biased answer picks a group the
  context doesn't justify.
- `jailbreak.jsonl` — 10 adversarial prompts (refusal expected) **+ 6 benign
  controls** (answer expected). The benign controls are what make over-refusal
  measurable — a model that refuses everything would otherwise look "safe."

**Judge.** `eval/judge.py` uses the configured frontier model (Gemini or Claude) at temperature 0 with strict per-axis JSON
rubrics. Each metric gets a Wilson 95% CI (`eval/metrics.py`) so small-sample
noise is visible rather than hidden.

> **Responsible-evaluation notes.**
> 1. The jailbreak prompts *request* disallowed content; they contain **no
>    harmful payloads** and exist only to test refusal behaviour.
> 2. The judge and one arm under test are the same model family — a **self-preference
>    bias** risk. It's documented in `judge.py`; to neutralise it for a real
>    submission, run the judge on a *different* model family (set `JUDGE_MODEL`)
>    or add a second judge and report agreement.

## Public deployment (bonus)

Two complementary deployments, both wired into the same project:

**Hugging Face Spaces — free CPU.** `deploy/hf_space/` is a self-contained
Gradio app serving `Qwen2.5-0.5B-Instruct` with the same prompt + guardrails as
the `oss_guarded` arm. Create a Gradio Space, upload `app.py` +
`requirements.txt`, done — public URL, $0.

**Modal — GPU, larger model.** `deploy/modal/modal_app.py` serves
`Qwen2.5-7B-Instruct` behind a vLLM **OpenAI-compatible** endpoint that
scales to zero when idle. The main project targets it with no code changes via
`OSS_BACKEND=endpoint`. `modal deploy deploy/modal/modal_app.py`.

## Cost & latency

**Default frontier (Gemini, free tier): $0.** Google AI Studio's free tier has no
marginal cost, so `observability.frontier_cost` returns $0 for `gemini-*` models.
This collapses the cost axis to "both options are effectively free here," leaving
**quality, safety, and latency** as the differentiators the report focuses on.

If you instead run the **Anthropic** path (`FRONTIER_VENDOR=anthropic`), cost is
computed from real token usage (`assistant/observability.py`, `FRONTIER_PRICING`,
USD per million tokens, verified 2026):

| Model | Input | Output |
|---|---|---|
| Claude Haiku 4.5 | $1 | $5 |
| Claude Sonnet 4.6 | $3 | $15 |
| Claude Opus 4.7 | $5 | $25 |

Prompt caching (−90% on cached input) and batch (−50%) are available levers for
production cost reduction. OSS cost is `latency × OSS_GPU_USD_PER_SEC` — **$0 on
local CPU / the free Space**, ~$0.0002/sec on a Modal A10G. For the OSS model,
**guardrails are the highest-ROI safety spend** because they cut jailbreak
success at zero marginal cost.

## Tradeoffs (OSS vs frontier)

- **Quality / risk.** The frontier model is far stronger on all three axes.
  The small OSS model hallucinates and stereotypes substantially more, and
  prompt-time guardrails **cannot fix that** — they only address the safety axis.
- **Cost / control.** OSS is ~free to run and fully self-hostable (data never
  leaves your infra); the frontier model has per-token cost and a vendor
  dependency, but no MLOps burden.
- **Latency.** A hosted frontier API was *faster* here than a small model on
  modest hardware — counterintuitive, and a reason to measure rather than assume.
- **Safety architecture.** Guardrails buy a large safety improvement cheaply but
  introduce **over-refusal** (a real UX cost), which is exactly why the harness
  measures both directions instead of only counting refusals as wins.

## What I'd improve with more time

- **Bigger / multiple OSS models** (7B, 32B) to map the quality-vs-cost curve,
  not just one point.
- **Independent judge + human spot-checks.** Remove the self-preference risk;
  report inter-judge agreement (Cohen's κ).
- **Larger, externally-sourced datasets** (full BBQ, TruthfulQA, a standard
  jailbreak suite) for tighter confidence intervals.
- **RAG / tool grounding** to attack hallucination at the source rather than
  only measuring it.
- **Streaming + semantic output guardrails** (an embedding/LLM classifier) to
  catch harms the regex layer misses, and reduce over-refusal with a smarter
  allow-list.
- **CI gate**: fail the build if any risk metric regresses past a threshold.
